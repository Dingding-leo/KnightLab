# Architecture

## Current vertical slice

KnightClub 0.2 uses:

- React 19 and strict TypeScript
- Vite and `vite-plugin-pwa`
- `chess.js` as the legal-rules authority
- A pure absolute-timestamp chess-clock state machine
- A pure player-side domain for resolved White/Black/Random bot ownership
- An original named-opponent profile domain with legal local opening cues
- A pure completion domain for resignation, draw agreement and timeout
- Lazy, optional Web Audio synthesis with no external audio assets
- A custom accessible board UI
- A Tauri 2 desktop shell with a Rust UCI process supervisor
- Stockfish for browser and desktop bot moves/MultiPV review, plus a bounded minimax failure fallback
- A normalized engine-settings domain plus native executable picker and UCI probe
- A fail-closed completed-review-to-retry adapter and due-first personal Train queue
- One-shot, latest-wins local-history Workers for retry items, saved games and tactics progress
- A one-shot, latest-wins active-session restoration Worker that revives verified `chess.js` snapshots without replaying a long PGN on the UI thread
- Versioned SQLite as the desktop authority, with bounded browser localStorage compatibility
- Vitest and Oxlint

The application contains no required runtime network requests after its static files are delivered. Website builds package checksum-verified Stockfish 18 Lite JavaScript/WebAssembly assets; the lightweight app shell installs first, then a narrow cache-first PWA route retains the optional engine pair after first use. The desktop app uses a separately installed local executable.

### Clock execution

`ClockState` stores each side's remaining time at an absolute `turnStartedAtMs` anchor. `ClockRuntime` owns one display-only timer and shares its snapshot only with the two player clocks; it stays mounted outside the Play tab so a real clock can still flag while a player reads Review or Train. The runtime wakes at the next visible whole-second boundary above 20 seconds and at the next tenth below it, records the exact timestamp for terminal persistence, and reports a flag exactly once without mutating clock state or writing storage. This keeps the App shell, board and setup panel out of low-time repaint work. When a started timed game is open in another workspace, the `LiveGameDock` is the other clock-context consumer: it provides both live values, the active/paused side and a return action without subscribing the App or board to each tick. The board then memoizes individual square wrappers with stable callbacks/refs, so selection, drag, premove and keyboard focus reconcile only affected squares instead of replacing all 64 buttons. A newly configured timed game is ready but unarmed, so neither clock charges until its first legal move. Legal human and accepted engine moves settle the mover, apply Fischer increment, reset delay and activate the opponent exactly once. Pause settles first, timeout cancels engine work and locks the board, and undo restores the settled pre-move clock with a fresh anchor.

### Player-side execution

`humanColor` is a resolved game role (`w` or `b`), while `orientation` remains a separate presentation preference. A bot setup request may be White, Black or Random; Random resolves only when a fresh game begins and stores both the request and the resolved side, so session restore cannot silently redraw it. The engine searches whenever the opposite side is to move, allowing Stockfish to make the opening move when the user plays Black. Before a bounded request, `chess.js` first identifies a rules-proven only legal reply; that local result preserves the usual response pacing while consuming no engine work. Input, clocks, player bars, draw/resignation attribution and whole-turn undo all use this resolved ownership. Opening a restart or resignation decision immediately invalidates an active bot request; cancelling a decision resumes a fresh search only for the unchanged position.

Native Play validates cancellation both before UCI setup and again after waiting for the single-engine mutex. Its one frontend client produces monotonic request IDs, so native stops retain the highest cancelled ID: a delayed older stop cannot revive a newer cancelled request. Any request at or below that watermark cannot create/reset a supervisor, reapply options, send `position` or issue `go`. Analysis clients share one renderer-wide increasing ID sequence but retain exact-ID cancellation rather than a numeric watermark. Native analysis consumes the matching marker before setup, after the engine mutex and between initialization, option setup, position and search work, then clears that exact marker on every completion path. This preserves the pre-cancelled queued-request fence without retaining ordinary navigation cancellations for the whole app.

### Named-opponent execution

`BotProfile` is original KnightClub data: name, monogram, bounded Stockfish target, presentation copy, a short opening-cue table and an explicit close-line preference. Choosing another opponent uses the existing restart gate, maps the profile to its established Easy/Balanced/Strong resource ceiling and persists the ID in the active session, completed game and preferences. On a bot turn, `selectProfileOpeningMove` requires the standard start FEN, exact SAN history, correct side to move and a successful `chess.js` application on a clone. Play supplies its already-cached SAN snapshot, so a long history is not rebuilt once per authored cue on the bot-turn hot path. A hit applies that legal authored move after the usual display pacing without constructing or searching an engine; if no cue applies, a rules-proven only legal reply also remains local before any bounded Stockfish request. `selectProfileCandidateMove` may then choose PV2 only when PV1 exactly matches Stockfish’s returned limited-strength `bestmove`, both lines have exact centipawn scores, the same-position loss is within the profile limit, the alternate move is legal on a cloned `chess.js` board and its locally-computable traits improve the declared preference. Mate, bound, stale, malformed and fallback candidates retain `bestmove`; no profile selection mutates the displayed game before final move application. The request retains the same time/node/thread/Hash limits and uses no extra `go`. If Stockfish fails, the UI identifies the generic KnightBot fallback rather than attributing its moves to a profile.

Rowan Pike is the default opponent, so its standard-start `1. e4 e5 2. Nf3 Nc6` route is intentionally covered by real local opening cues. This avoids booting the browser’s Stockfish WebAssembly runtime for the common first two replies while preserving a bounded engine request immediately when the player diverges from that route.

### Premove execution

During a local bot turn, one human premove may be queued without restarting the active Stockfish request. Its preview intentionally checks only source ownership, target ownership and plausible piece geometry: the pending bot move can make an en-passant capture, open a line or remove a check. When the accepted bot result arrives, the application clears the queue and asks a cloned `chess.js` position to apply it as the sole legality authority. A rejected premove leaves the bot position intact and reports the cancellation. A successful premove completes both clock moves at the same timestamp, so it consumes no human time but receives the normal increment; both settled snapshots are retained for undo. Queues are visual-only, never persisted, and clear on any pause, restart, terminal outcome, undo, position load or hydration.

### Play-flow safeguards

New game, time-control, mode, FEN-load and saved-game actions share one restart gate. It asks for confirmation only when replacing an unfinished game, pauses the current clock while the decision is open, and restores that clock on cancel. The board keeps click/tap and keyboard selection while Pointer Events with capture make drag-to-move work for mouse, touch and pen without a separate HTML drag implementation. Outside that roving-focus grid, an unmodified Left/Right Arrow moves through Play's existing non-live preview states; an editable control, modal, modifier or board-grid target cannot invoke it, and reaching newest returns to live. A promotion dialog is its own keyboard boundary: it initially focuses Queen when available, exposes Q/R/B/N through `aria-keyshortcuts`, accepts only an offered unmodified piece key, and keeps Escape as a cancel action; unrelated Play shortcuts remain blocked behind it.

Play setup uses progressive disclosure rather than permanently placing configuration ahead of the current game. A zero-ply session starts with the native setup disclosure open; the first recorded ply closes it once, leaving an original compact opponent/side/time summary, accessible Draw/Resign actions and notation above the fold. Restarting a fresh game opens it again, while a user who later reopens it keeps it open until another fresh session begins. The disclosure body is unmounted while closed, so inactive opponent, engine and time controls do not participate in ordinary in-game rendering.

The custom-time form keeps its uncommitted minute, increment and delay text in a persistent ref-backed draft. Its native inputs therefore update themselves while typing rather than scheduling a root `App` render; closing and reopening the mounted form restores that draft, and clicking **Use custom time** remains the single validation/restart boundary. A restored custom control seeds the same fields from its persisted milliseconds so players can adjust the actual saved values.

Every immutable Play `Chess` state exposes one verbose history snapshot. The SAN move list derives from that snapshot, and human commits, bot replies and undo extend or trim it from their already known legal move instead of asking `chess.js` to rediscover the whole list. `MoveList` groups the immutable SANs into memoized rows with primitive current/latest flags; one stable delegated click handler reads each button's validated ply, so a live append changes only the old/new latest row and an historical selection changes only its affected row. A memoized `gameSummary` supplies the complete board-terminal state, result and wording from one ordered rules pass, while material is memoized by the displayed live/preview board; selection, premove and notice renders therefore do not repeat terminal legal-move generation. A current-position copy first uses the platform deep-snapshot primitive and verifies its FEN; the existing start-FEN replay remains the safe fallback when that primitive is unavailable. This preserves full undo/repetition/PGN semantics rather than cloning only the final FEN. Historical preview is explicitly read-only, so it constructs a new board from the selected verbose move’s exact `after` FEN rather than replaying any prefix; malformed supplied FEN falls through to the established bounded prefix replay. The current-game PGN is likewise memoized once per position and shared by transfer and latest-wins session persistence; terminal export still rebuilds its result-aware headers deliberately.

#### Live PGN path

Play passes the same immutable verbose move snapshot to its PGN serializer that it already uses for notation, review-key creation and move commits. The serializer emits the standard `chess.js` header/movetext shape directly from SAN, side, and pre-move FEN facts, so a normal move no longer makes `chess.js` undo and replay all prior moves before browser/SQLite active-session persistence or local copy/download. For chess.js's known plain, empty private comment store it also skips `getComments()`: that public API prunes comments by undoing and replaying the complete history even when no comment exists. Setup FENs, Black-start numbering, promotions and explicit Result/Termination headers stay standards-compatible. A game carrying imported comments, or an unfamiliar comment-store shape, falls back to the authoritative `chess.js` serializer to retain annotations exactly; that uncommon path is intentionally allowed to pay the full replay cost.
Terminal result/termination export uses this same direct path with its explicit header overrides rather than reintroducing a full history replay.

### Workspace-navigation handoff

A primary-workspace change is a navigation handoff, not a game-state action. After React has rendered a different Play, Review, Train or Library workspace, the shared handoff contract moves the window to the top and places programmatic focus on the current `#workspace-title` with `preventScroll`. The `<main>` landmark is named by that same title through `aria-labelledby="workspace-title"`, while the heading's `tabIndex={-1}` keeps it out of the ordinary tab sequence.

Activating the already selected workspace is deliberately a no-op: it neither scrolls nor moves focus. This preserves a reader's place while a real workspace change gives pointer, keyboard and screen-reader users a stable beginning and landmark name.

A Review → Train practice target is a one-visit navigation payload. It remains available through that mounted Train session, including an optional replay of an already-mastered item, and is cleared only after the player leaves Train. A later visit therefore returns to the ordinary due-first personal queue instead of silently re-opening an old target.

Play-critical board, clock and game controls remain in the entry bundle. Review, Train and Insights each live behind an independent React lazy boundary so their review/training/dashboard modules do not delay first board paint. Hovering or focusing one of those navigation buttons starts only that module fetch before activation; a sized loading card preserves the page shape without taking focus from `#workspace-title`. A boundary local to each workspace presents an honest Reload action if a stale or unavailable async PWA asset fails, rather than taking down the Play shell. On Review mount, one safe PGN parse supplies both the immutable timeline and initial cursor; its live-continuation check reuses that same timeline while the normalized current PGN is unchanged, parsing again only for a genuinely changed game.

The first Play render deliberately does not read the browser retry, game-library or tactics-history keys. On a Library/Insights or Train intent, the renderer makes only an opaque localStorage read, sends that raw text to a one-shot latest-wins Worker and rechecks the raw value before adoption. Library adoption projects each validated game to metadata without its PGN, so React state, filtering and Insights do not retain a complete personal move archive. A selected Open or Review action asks the same Worker for exactly one full record. The Worker and yielded fallback share pure bounded, fail-closed parsers; an unavailable Worker therefore cannot make first paint expensive. Library keeps a retryable error state rather than clearing an in-memory game, while Train holds the relevant panel behind an explicit local-preparation status. A deferred personal queue restores the due-first panel only if the player has not deliberately selected another trainer during preparation.

Active-session recovery follows the same paint-first rule. A browser payload over 2,048 characters is not JSON-parsed during render: Play starts in a visibly locked recovery shell and a one-shot Worker validates its 512 KiB PGN bound, replays the game and prepares verbose notation. Small bounded sessions retain the immediate path. The Worker returns a structured-cloneable own-property snapshot of `Chess`, the final public FEN, the private history depth and the precomputed verbose history. The renderer reattaches the `Chess` prototype, verifies FEN/history depth and supplies the existing verbose list to Play, preserving undo, threefold repetition, headers/comments and exact notation without a UI-thread replay. A raw-storage freshness loop retries a newer mirror after either a response or a Worker error, but stops after three changes; malformed, over-limit or continuously changing data remains behind an explicit **Start fresh instead** choice rather than being cleared or overwritten. Until adoption, board/setup input, shortcuts, bot work and active-session persistence are blocked so the fallback state cannot overwrite the real game. On desktop, SQLite bootstrap is authoritative from the first render; each non-null bootstrap session and an empty-database browser-mirror migration use the same Worker boundary. A fresh-start choice owns a separate bootstrap generation: it suppresses a late stale session import, hydration and adoption while allowing preferences/library migration to finish, then queues one final native clear after those writes so the rejected game cannot return on a later launch.

An explicit Library **Open board** uses that same verified chess-snapshot machinery through a distinct `hydrate-stored-game` request and its own client, so it can never cancel startup recovery. A selected PGN above 2,048 characters first commits a Play overlay; only then does the Worker replay it and create the canonical review key from the already-extracted verbose moves. The old board/clock remain paused and the board and setup panels are `inert`, while the overlay retains focus on **Cancel opening** outside their busy regions. A request version fences every success/error/cleanup response. Cancelling, leaving Play or an error flushes the paused predecessor snapshot before reopening it; successful verified adoption discards that fallback and lets normal active-session persistence schedule the replacement. A long selected record never falls back to synchronous main-thread replay if a Worker is unavailable—its old board remains intact with a recoverable error. Browser Library records remain character-bounded for legacy storage, so the Worker permits their worst-case four-byte UTF-8 representation; native SQLite continues to enforce its tighter byte envelope.

### Completion execution

`GameTermination` is the single serializable authority for non-board endings: timeout, resignation and draw agreement. Opening a destructive confirmation or hot-seat draw response settles and pauses the clock, which also cancels active engine work through the existing effect lifecycle. Decline/cancel resumes the same settled clock; confirmation locks the board and saves one result-aware PGN. Bot draw decisions are deterministic from material, game phase and strength until full engine evaluation is available.

### Fallback bot execution

The built-in fallback bot is deliberately separated from React through a typed worker protocol. It is created only when the browser or native Stockfish runtime cannot complete a bot search. `BotWorkerClient` permits only one active search, terminates the worker when a search is superseded, recreates a clean worker, and accepts a result only when both request ID and FEN match. Its one-ply, node-bounded search is an outage safety net, not a second high-CPU engine.

  ├── DatabaseClient + localStorage compatibility repositories

        ├── StockfishClient (desktop) ── typed Tauri command ──┐

  ├── DatabaseState

`chess.js` remains the only legal-rules authority in the UI. The engine proposes a UCI move; the current FEN and request ID must match, UCI syntax is validated, and `chess.js` applies the move. This keeps engine output outside the trusted game-state boundary.

### Review analysis execution

The frontend builds a complete immutable position timeline from bounded PGN/FEN input using `chess.js`. Small initial games retain an immediate Review handoff; longer initial games plus pasted/file imports use `TimelineWorkerClient`, a latest-wins dedicated Worker that runs the exact same parser and returns a complete immutable timeline only after the loading shell has painted. Superseding imports, source handoffs and unmounts terminate a stale replay rather than waiting for it to finish. Live-game continuation comparison uses the same background path, so a long active PGN cannot freeze cursor navigation. Selecting a ply updates the board immediately, cancels an old request and waits for a 350 ms idle window before submitting the exact current FEN to an isolated analysis client. On desktop, Rust configures full-strength native Stockfish and returns typed MultiPV data. In the website, a dedicated Worker runs Stockfish 18 Lite WebAssembly and the TypeScript UCI adapter parses the same score, WDL, PV and search metrics. The UI validates the response and derives SAN without trusting engine text as game state. Exact scores are retained over later bound-only updates.

Interactive Review keeps a session-local 24-entry LRU only for those ambient candidate lines. Its key contains the runtime kind, configured desktop executable path when applicable, exact FEN and normalized time/depth/node/MultiPV/thread/Hash settings. A hit returns an independent copy immediately, labelled as a cached local result, and avoids both the debounce and a new UCI `go`; backend, path, position or setting changes miss. Only a response that has already passed client validation and SAN conversion is inserted. Errors, aborts and every full-game-review search stay outside this cache, so reproducible review jobs retain their own sequential engine contract.

`runGameReview` composes that same strict client into a sequential, abortable job. It first rejects a timeline above the shared 1,024-ply persisted-review bound, before creating any engine request; the UI expresses the same limit and retains normal navigation plus per-position analysis for those long imports. It searches the start and each intermediate position once with the required MultiPV baseline, then uses the first line of that next-position result as the previous move's after evaluation. The final non-terminal position keeps the cheaper single-line after check; checkmate and draw positions stay in `chess.js`. Scores are normalized to the mover before immutable inputs reach the pure review model. After each fully classified non-final ply, `runResumableGameReview` emits a copied session-only checkpoint: its report is contiguous through that ply and its raw next-position MultiPV response becomes the next baseline on Resume, avoiding a duplicate search. Resume validates the source-move prefix, completed/total ply, baseline FEN, effective bounded settings and existing engine name/path; a newly returned engine identity must still match before another move is classified. The React workspace keeps that checkpoint separate from a completed `GameReview`, so provisional scorecards/feedback never trigger persistence, Retry verification or Training. **Stop & keep** aborts the active client but preserves the checkpoint; **Resume review** fixes the original settings, while **Discard** removes only session memory. Imports, source handoffs and unmount clear it. While paused, ambient candidate analysis stays off, ensuring that a user who stops review really returns CPU to the device. UI progress is monotonic by completed ply; Stop cancels the active native or browser search. Starting a browser job disposes the idle ambient client and releases an idle Play browser runtime before the sequential client is created, so a full review does not retain unnecessary Stockfish WebAssembly Workers or hash allocations; completion, Stop and timeline/navigation changes release that job client. Identity-guarded cleanup prevents a late old job from disposing a newer one. A bot turn or manual Verify has product-level priority: Review's ambient search and full-review action wait for it instead of competing for local CPU.

The ordinary replay board receives one shared inert interaction object. **Explore this position** creates a local `VariationState` with the selected immutable FEN and source ply as its anchor. Its compact, canonical move facts are strictly replayed through `chess.js` whenever an untrusted, cloned or stale public state reaches the boundary, so forged or illegal branch state fails closed. Normal sequential states retain a private identity-keyed, signature-checked `chess.js` history instead, avoiding a full legal replay of every earlier branch ply and deferring PGN serialization until the user actually copies or downloads it. Only an active branch swaps in click/drag callbacks and legal target markers; promotion always pauses for an explicit q/r/b/n choice, focuses its first option and blocks Review navigation keys until it is resolved. The branch's current FEN—not the main-line cursor—feeds the board, last-move marker, candidate-line cache key and FEN/PGN transfer. Coach evidence and selected-move retry actions stay hidden during a branch because they describe the main line. Every main-line navigation route, import, requested handoff and full-review action clears the branch; return restores its anchor ply. The branch is intentionally session-only and has no database/localStorage representation. The long notation list uses memoized rows plus one delegated `data-ply` handler, initially mounts the newest 40 rows and reveals earlier rows in explicit 40-row steps; an early selected row stays pinned without remounting the omitted middle. It is keyed by the canonical review identity, so importing a different long game resets the bounded window instead of carrying an earlier expansion across. The row lookup uses the actual source ply, so a setup FEN that begins with Black does not mis-pin the next White move. For normal timelines the mobile picker isolates its complete immutable option set from its changing value; over 1,024 plies it instead renders a bounded numeric Go field, avoiding a giant hidden option tree. Full-review progress emits before/after updates for every ply, but those updates now repaint only the progress surface unless the selected ply, timeline, evidence or completed review actually changes; ordinary replay navigation updates only its old/new notation rows. The initial long-PGN Worker-routing decision is fixed for the mounted workspace, so those progress renders do not repeatedly UTF-8 scan the unchanged PGN. The visual bar updates continuously, while a separate polite status announces only start, five-percent milestones and completion rather than queuing every before/after update. Selected review metadata is indexed directly by its contiguous persisted ply instead of scanning the full report on every replay step.

After a completed PGN review, the frontend creates a deterministic review key from canonical start FEN and main-line UCI moves, validates a bounded versioned record and saves it before showing a retained status. Timelines above that record's 1,024-ply limit skip the key/hash and hydration lookup entirely because they cannot become a completed report. New terminal Library games derive that identity directly from their existing verbose `chess.js` history. A saved-game **Review** keeps Play untouched, mounts Review with the selected PGN and lets the long-PGN Worker path paint its shell before replay; a destructive **Open board** defers selected-PGN parsing until after its confirmation. `ReviewStorage` uses desktop SQLite after database hydration or bounded browser localStorage otherwise. Browser localStorage keeps a private raw-text-versioned envelope index: a new write and the requested same-key candidate receive the full fail-closed PGN/report replay, while unrelated retained reports do not. A corrupt newer duplicate falls through to an older valid same-key candidate, and any direct storage rewrite invalidates the index. Hydration is versioned so an asynchronous result for an older import cannot replace the active timeline. While a matching saved-report lookup is pending, the full-review action is disabled so it cannot invalidate the lookup and spend a duplicate engine pass; after a report is present, the explicit action reads **Review again**. A storage failure leaves the completed result visible with an explicit non-retention error. Matching Library records are found by a single metadata-only key pass; source metadata can be linked after completion without replaying unrelated PGNs.

Coach evidence is derived synchronously from an already completed review; it never starts another Coach/review engine request or mutates the saved report. The pure coach domain reconstructs the reviewed move's exact pre- and post-move FENs, validates the recorded best UCI move with `chess.js`, and may expose only directly provable facts: a mating move, a check, an unsupported moved piece, a direct double attack, or an absolute king pin. Limited-confidence, malformed or unprovable cases degrade to a neutral comparison with the recorded SAN line. The React card projects its verified squares into a presentation-only board highlight and the Review workspace owns scoped replay-key navigation; both remain derived state, so restored reports behave identically. Ordinary position analysis may still follow the user-selected replay ply when it is enabled.

### Local PGN/FEN transfer boundary

Play exposes the current game's PGN through explicit copy and plaintext-download actions, and exposes the current FEN through the Position tools. Its success or recoverable-error status is rendered beside the Play toolbar. The transfer adapter attempts the platform Clipboard API first; if a browser or WebView rejects it, it tries a selected temporary textarea copy path and returns a user-facing result rather than letting a rejected promise escape. Downloads are generated from local plaintext `Blob` object URLs and are released after the browser receives the click.

Review continues to accept pasted PGN/FEN and may read a user-selected local `.pgn`, `.fen` or `.txt` file through a visible keyboard-focusable picker button. Selection is not an upload and does not write a Library record. The picker rejects a declared file over 512 KiB before calling `File.text()`, then the file boundary treats both the reported size and actual UTF-8 text as untrusted; a FEN is additionally capped at 1 KiB. Extension/content detection is only a parser hint: a valid PGN misnamed `.fen` may fall through to PGN parsing, while `chess.js` timeline creation remains the notation authority. A request gate permits only the newest pending file selection to apply. No file or pasted value is applied until parsing produces an immutable timeline, so file-read, size, notation or parser failures preserve the prior Review timeline.

### Personal retry execution

The retry adapter consumes only a finished review, its immutable timeline and optional already-derived Coach focus. Its public boundary replays the canonical timeline with `chess.js`, confirms that the selected normal-confidence adverse move belongs to that source ply and that both the played and recorded solution UCI moves are legal from the exact pre-move FEN. For one active Review report, the UI may first create a private frozen snapshot of that same complete validation; selected-ply and small batch prompt creation then read its detached verified facts in O(1). A module-private capability registry rejects structural lookalikes, and the standalone public adapter retains the full replay contract for untrusted callers. It snapshots a bounded `solutionLineSan` only from the completed review's saved Stockfish principal variation (`bestLineSan`), never from the original PGN continuation after the error. Any source mismatch creates no prompt. Review can save one selected prompt or a small ranked batch; Train reconstructs the stored FEN and orients the board to the colour that made the reviewed move.

Before exposing a Train line, the retry-line adapter reconstructs every saved SAN ply locally with `chess.js`, checks its canonical UCI/SAN representation and confirms that its first ply is the saved solution. An explicitly empty legacy `solutionLineSan` safely uses that independently verified first move; a non-empty stored line that cannot replay in full is unavailable rather than shortened or guessed. No retry action invokes Stockfish or changes the saved report. Until a user asks for a hint/reveal or makes an attempt, Train holds back the recorded answer, future PV plies and Coach evidence. For a valid multi-ply PV, Train accepts only the reviewed player's next exact move, auto-applies the next recorded opponent PV reply, then prompts for the next player move. A legal alternative is only reported as not being on the saved review line; alternatives are not scored.

The due-first picker deliberately materializes only the current item's validated retry line. Its selector labels use the durable FEN fullmove/side facts (with a legacy source-ply fallback), so unopened items do not replay `chess.js` solely to show a move number. It initially mounts 24 positions and adds another 24 only on **Show more**; the current or review-requested item is reserved in the visible subset without changing canonical due-first order. Phone picker and expansion controls retain 44 px touch targets. Browser retry persistence has a private `WeakMap` snapshot keyed by storage identity and exact raw text: first load and every external raw change fully parse and fail-closed validate the bounded queue, while an ordinary save/load-one/delete only validates the changed item and updates the canonical sorted snapshot. Public values are schema-projected copies, so unknown legacy nested fields and caller mutations cannot contaminate the cache.

Only a full unassisted saved-line replay advances the five-step 1/3/7/14/30-day schedule, and it writes that success once at the terminal line state. A non-matching legal attempt, hint-assisted completion, reveal or skip resets the streak and keeps the prompt due immediately. The in-progress board, cursor, selection and hint state are transient React session state; the self-contained retry item retains its durable prompt and schedule independently of completed-review retention. **Back to review** loads the retained source PGN and focuses its recorded ply; if a bounded old report was pruned, the retry remains usable but the source-review return clearly reports that it is unavailable.

Production browser builds register the PWA service worker; the app shell precaches independently from the optional Stockfish Worker/WebAssembly route, which CacheFirst stores only after use. Vite development and Tauri deliberately do not register it. Tauri startup removes any service worker/cache left by older builds so an upgraded desktop package cannot continue serving stale UI from `tauri://localhost`.

### Engine settings execution

`EngineSettings` is normalized before persistence and again before every frontend request. Preset profiles remain adapter-owned; Elo and Custom profiles send bounded skill, move-time, depth, node, MultiPV, thread and Hash values. Interactive drafts are validated before state changes and receive field-specific accessible feedback when rejected; persistence and adapter normalizers then independently return malformed, fractional or out-of-range values to safe defaults rather than clipping them to a resource maximum. Rust independently rejects desktop values before emitting any UCI command. At the live-Play boundary, both adapters and Rust separately preserve a profile's strength identity but cap every reply to its Easy/Balanced/Strong 50/50/60 ms and 1k/1.5k/3k node ceiling, one thread, 16 MB Hash, no depth cap and one base PV; a named profile may raise only that same command to two PVs. Review keeps its separately disclosed analysis contract. The browser adapter enforces its single-threaded build and caps Hash at 128 MB, while its direct analysis boundary also restores invalid request values to its ordinary bounded analysis defaults. Each persistent browser Worker or native supervisor keeps its own last option vector only after `readyok`; unchanged settings skip the redundant `setoption` block but each search still sends `isready` before its new position/go pair. A normal cancellation sends `stop`, preserves the warm runtime/vector and lets that next readiness fence drain delayed output; any worker/process drop, timeout or failed setup clears the cache. A settled full desktop Review then requests a non-blocking idle release: the shared pool drops its process only when `try_lock` proves no newer Play or Review owns it, returning high Hash/NNUE memory without interrupting a live request. A browser probe is gated while a live bot move owns its local worker. On desktop, a probe locks and reuses Play's managed supervisor rather than spawning a second UCI process; a racing Verify waits for the bounded Play request, then checks the same warm runtime. Controls disclose that single-engine-task priority, and choosing a new path or changing search settings invalidates the current frontend search through the existing effect cleanup.

The same idle predicate is shared by browser and desktop Play: only after the player has left Play or the game has finished, and only without a bot reply, premove or Verify task, browser Workers are disposed or desktop asks the shared pool to `try_lock` and release its idle child. The desktop command is deliberately non-blocking, so a concurrent or newly acquired Play/Review task keeps its native process rather than being interrupted.

## Engine isolation rules

- Keep Stockfish outside KnightClub's original source and communicate only through UCI text streams.
- Run the website engine in a dedicated Worker; run the desktop engine as a separately managed process.
- Pin and verify every distributed engine asset, include GPLv3, and publish the exact corresponding source location beside it.
- Record version, checksum/runtime, source URL and engine settings when analysis caching is introduced.
- Reject stale responses using request identifiers and position hashes.
- Implement stop, timeout, restart, bounded queues and process cleanup.
- Never block the UI thread.

## Failure model

- Discovery or Worker failure: report the error for verification and use the isolated KnightBot fallback for bot play.
- Invalid input/output: reject it without changing the board.
- Cancellation: frontend invalidates the request immediately; Rust observes an atomic marker or the browser adapter sends `stop` and drains the old `bestmove`.
- Timeout, process exit or an unresponsive Worker: discard that engine runtime so the next request starts cleanly.
- Late result: reject unless both request ID and FEN equal the active search.
- Flag fall: settle to zero, invalidate engine work, persist the timeout result and reject all later moves.
- Completion dialog: pause before the decision; on cancel resume from the settled anchor, on confirmation persist and reject later moves.

## Persistence execution

Desktop startup opens `knightclub.sqlite3` in the Tauri application-data directory, runs `quick_check`, applies forward-only `PRAGMA user_version` migrations and then exposes only task-specific commands. Schema v5 stores singleton active-session/preferences JSON, indexed completed-game rows, a separate bounded `reviews` repository, a bounded self-contained `retry_items` repository, and bounded tactics progress/immutable-attempt records. Its startup bootstrap returns only session, preferences, recovery state, a bounded game count and whether all persistence tables are empty; it deliberately does not deserialize game PGNs. A player opening Library or Insights requests validated bounded summary rows with `pgn` removed; only a selected row calls the validated full-detail command. That delayed response is queued behind earlier writes, merged under current in-memory game updates and invalidated by Clear, so a just-saved/reviewed game cannot disappear and a pre-clear response cannot resurrect deleted rows. If every persistence table is empty, the frontend imports bounded legacy localStorage state in one transaction; that one-time migration deliberately uses the full Worker parser, while ordinary Library/Insights state remains summary-only. An existing database remains authoritative. Browser and native active-session writes share a latest-wins 250 ms deferred lane so a move never synchronously serializes a long PGN/clock history; a terminal result, page hide or visibility loss flushes the newest snapshot, while restart/clear discards the old pending snapshot before the existing FIFO native clear barrier. Once execution begins, the native client retains its own contiguous-snapshot coalescing and FIFO ordering, so no ordinary write or clear can be crossed.

Library view filtering occurs over the complete validated local summary set before pagination. The UI mounts an initial 24-card page and reveals another 24 only when requested, resetting to the first page on a search/filter/aborted-game change but not when a background library merge arrives. A per-record weak cache avoids recreating date/profile search text across successive query keystrokes; rendered date formatting is likewise limited to the current page. A row shows **Opening…** while its single detail request is in flight, and a new request supersedes the old one.

Malformed or oversized payloads are rejected independently in TypeScript and Rust. Review payloads additionally have their PGN-derived identity, canonical FEN and a one-to-one per-ply match between the saved report and its source timeline checked at the frontend boundary; native code independently rejects absent, incomplete or non-contiguous review move lists before a record is saved or restored. Retry records use a separate item schema, bounded metadata and due-first index; TypeScript additionally reconstructs the FEN and validates the played/solution move facts before they enter UI state. A non-empty stored `solutionLineSan` must also reconstruct completely as a canonical local PV; only an explicitly empty legacy field may use the verified first-move fallback. A database that fails integrity checking is renamed to a timestamped `.corrupt-*.bak` file (including WAL/SHM sidecars) before a clean schema is created; the Library reports the recovery path. Browser/PWA builds keep bounded localStorage compatibility repositories for games, completed reviews and retry items. Later phases still need duplicate PGN hashes, manual backup/restore, and indexed position, opening and tag queries.

## Trust boundaries

Untrusted inputs include PGN, FEN, imported datasets, engine output, file paths, persisted engine settings, retry prompts and database backups. Parse them defensively, bound their size, reject malformed records and never interpolate them into shell commands. Engine settings are bounded independently in TypeScript and Rust before they can become UCI commands.
